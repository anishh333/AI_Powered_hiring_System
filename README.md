# AI Resume Evaluator

An end-to-end resume-vs-job-description evaluator with an explainable match
score and a conversational candidate-search assistant, built on **local,
non–Hugging Face** ML (scikit-learn TF-IDF) so there are no model downloads
and no external API required to run it.

## What changed from the old version

- **PDF-only inputs.** The evaluator tab now takes a JD PDF and a Resume PDF
  (via `pypdf`) instead of pasted text.
- **No Hugging Face / sentence-transformers.** Semantic matching now uses a
  TF-IDF vector-space model (`scikit-learn`) fit on the local resume
  database — fully offline, no model downloads.
- **"Learning & System Metrics" tab removed.** That tab also referenced a
  hardcoded Windows path (`d:\CEI_FinalProject\...`) that would crash on any
  other machine — it's gone along with the tab.
- **Clean, simplified UI** — fewer nested custom HTML blocks, consistent
  cards, a score gauge, and skill "pills" for matching/missing/suggested
  skills.
- **RAG guardrail.** The conversational assistant (new "Ask the Assistant"
  tab) only answers resume/candidate/hiring-related questions. Off-topic or
  random input gets a clear "cannot process this request" style message
  instead of a hallucinated answer.

```
resume_evaluator/
├── app.py                  # Streamlit app (2 tabs: Evaluate, Chat)
├── requirements.txt
├── data/
│   └── resume_database.sqlite   # candidate resumes used for RAG search
└── src/
    ├── pdf_utils.py         # PDF -> text (pypdf)
    ├── text_processing.py   # cleaning + curated skill-vocabulary extraction
    ├── matching.py           # TF-IDF semantic similarity + skill overlap score
    ├── feedback.py           # explainable feedback (heuristic, or LLM-assisted if you add an API key)
    └── rag_search.py         # candidate search + off-topic guardrail
```

## Run it

```bash
pip install -r requirements.txt
streamlit run app.py
```

Then open the URL Streamlit prints (usually `http://localhost:8501`).

## How the scoring works

1. **Skill overlap** — a curated skill vocabulary (languages, frameworks,
   cloud/DevOps, data/ML, practices, soft skills) is matched against both the
   JD and resume text.
2. **Semantic similarity** — TF-IDF cosine similarity between the JD and
   resume, using a vectorizer fit on the local resume database so IDF weights
   reflect real hiring-domain language.
3. **Final score** = `0.6 * skill_overlap + 0.4 * semantic_similarity`.
4. **"Also worth having" skills** are retrieved (RAG) from the most similar
   resumes already in the database, so feedback isn't limited to whatever the
   JD happened to spell out.

## The conversational assistant

The "Ask the Assistant" tab retrieves candidates from `data/resume_database.sqlite`
using the same TF-IDF approach. Before answering, it runs a relevance check:

- If the query mentions hiring-domain words (resume, candidate, developer,
  skill, a known role/category, a known skill, etc.) → it searches and
  answers.
- Otherwise it falls back to corpus similarity, but only trusts that score if
  enough of the query's words are real vocabulary matches (to avoid one
  coincidental word producing a false positive).
- Common general-knowledge topics (weather, jokes, recipes, movies, etc.) are
  explicitly rejected even if a stray word overlaps with the corpus.

If a query doesn't pass, the assistant replies that it can't process the
request and asks the user to rephrase or ask something else — it does not
attempt to answer outside its resume/hiring scope.

## Optional: more natural-language feedback

By default, feedback summaries are generated by a local rule-based writer
(`generate_heuristic_feedback`) — no API key needed. If you add an OpenAI API
key in the sidebar, the same retrieved facts (score, matching/missing skills,
commonly-expected skills) are passed to an LLM for a more natural summary;
this is optional and the app is fully functional without it.

## Notes on the resume database

`data/resume_database.sqlite` contains 3,500 sample resumes across 36 job
categories, used only for the candidate-search assistant and for retrieving
"commonly expected skills" context. It is not required for the core
JD-vs-resume PDF evaluation, which works on the two uploaded PDFs alone.
